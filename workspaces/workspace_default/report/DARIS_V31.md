### 步骤3：主题管理脚本开发（严格按要求，仅4个文件）
所有脚本均存放于`scripts/`目录，使用Python标准库开发，无额外依赖，独立可执行，中文注释完整，异常处理完善，**永久移除所有Git分支相关参数与逻辑**。

#### 1. init_workspace.py（初始化新主题）
严格按照以下逻辑开发：
- **输入参数**：
  - 必选位置参数：`workspace_id`（主题唯一ID，仅支持字母、数字、下划线，全局不可重复）
  - 可选参数：`--name`（主题中文名称）、`--desc`（主题详细描述）、`--template`（配置模板，默认default）、`--no-switch`（创建后不切换为当前激活主题）
- **执行逻辑**：
  1.  参数合法性校验：校验`workspace_id`命名规范，检查同名主题是否已存在，存在则直接报错退出，避免覆盖数据；校验公共模板库是否存在。
  2.  创建主题专属隔离目录树，完整结构如下：
      ```
      workspace_{workspace_id}/
      ├── config/
      ├── data/
      ├── literature/
      │   ├── pdf/
      │   └── structured/
      ├── hypothesis/
      ├── code/
      ├── experiment/
      ├── memory/
      ├── knowledge_base/
      ├── logs/
      ├── report/
      └── workspace_meta.json
      ```
  3.  初始化主题配置：从`core/config_templates/`复制所有配置模板到该主题`config/`目录；生成专属`research_definition.md`，自动填充主题ID、名称、描述，预留关键词、研究目标等配置项；复制双记忆库模板到`memory/`目录，初始化空结构。
  4.  生成主题元数据文件`workspace_meta.json`，记录主题ID、中文名称、描述、创建时间、状态、模板类型。
  5.  更新全局主题索引`workspace_index.json`，新增该主题元数据，标记状态为`active`。
  6.  若未指定`--no-switch`：更新`.current_workspace`全局锚点文件，写入`workspace_{workspace_id}`，设为当前激活主题。
  7.  完整性校验：校验所有目录、配置文件是否创建成功，输出创建结果。
- **异常处理**：目录创建/模板复制失败自动回滚，删除已创建的无效文件，报错退出；同名主题已存在直接报错，不做任何修改。

#### 2. switch_workspace.py（切换激活主题）
严格按照以下逻辑开发：
- **输入参数**：
  - 必选位置参数：`workspace_id`（目标主题ID）
  - 可选参数：`--force`（强制切换，忽略配置完整性校验）
- **执行逻辑**：
  1.  校验目标主题目录是否存在，不存在则报错退出。
  2.  若未指定`--force`：校验目标主题的目录结构完整性、核心配置文件（research_definition.md、search_rules.yaml等）是否存在，校验不通过则报错，明确提示缺失内容。
  3.  备份原锚点文件为`.current_workspace.bak`，避免写入失败丢失数据。
  4.  写入新的主题目录名`workspace_{workspace_id}`到`.current_workspace`，完成主题切换。
  5.  更新全局主题索引`workspace_index.json`，将目标主题设为`active`，其他主题设为`inactive`，更新`current_active`字段。
  6.  切换结果校验：重新读取锚点文件确认写入正确，读取目标主题核心配置确认可正常解析，输出切换成功报告，包含原主题、新主题、主题名称、核心配置路径。
- **异常处理**：锚点写入失败自动从备份恢复，报错退出；目标主题不存在/校验不通过直接报错，不修改任何配置。

#### 3. archive_workspace.py（归档已完成主题）
严格按照以下逻辑开发：
- **输入参数**：
  - 必选位置参数：`workspace_id`（要归档的主题ID）
  - 可选参数：`--keep-source`（归档后保留原工作区目录）、`--output-path`（归档包输出路径，默认archives/）、`--password`（归档包密码）
- **执行逻辑**：
  1.  校验目标主题是否存在，若为当前激活主题则直接报错退出（禁止归档正在使用的主题）。
  2.  创建`archives/`目录（若不存在），更新主题元数据，标记状态为`archived`，记录归档时间。
  3.  更新全局主题索引，标记该主题为`archived`，从激活列表移除。
  4.  将主题目录完整压缩为zip包，命名规范为`workspace_{workspace_id}_{归档时间}.zip`，支持密码加密。
  5.  压缩完成后校验包完整性，若未指定`--keep-source`则删除原工作区目录，若指定则将原目录设为只读权限。
  6.  输出归档成功报告，包含归档包路径、文件大小、归档时间、源文件处理状态。
- **异常处理**：压缩失败自动删除损坏包，恢复主题状态为`active`，报错退出；归档当前激活主题直接报错，不执行任何操作。

#### 4. delete_workspace.py（删除无效主题）
严格按照以下逻辑开发：
- **输入参数**：
  - 必选位置参数：`workspace_id`（要删除的主题ID）
  - 可选参数：`--force`（强制删除无需二次确认）、`--include-archive`（同时删除对应归档包）
- **执行逻辑**：
  1.  校验目标主题是否存在，若为当前激活主题则直接报错退出（禁止删除正在使用的主题）。
  2.  若未指定`--force`，控制台弹出二次确认提示，用户确认后再执行删除。
  3.  彻底删除主题工作区目录，若指定`--include-archive`则同时删除archives/目录下对应主题的所有归档包。
  4.  更新全局主题索引，移除该主题的所有记录。
  5.  输出删除成功报告，确认目录、归档包已完全删除，索引已更新。
- **异常处理**：删除当前激活主题直接报错；删除失败输出错误详情，不修改索引。

**执行要求**：所有脚本开发完成后，逐行校验语法正确性，输出脚本开发完成报告，标注每个脚本的路径、功能、参数说明。

### 步骤4：核心路径解析模块开发（隔离机制落地核心）
在`core/utils/`目录下创建`path_resolver.py`，这是主工作流所有路径操作的**唯一入口**，强制锁死所有读写路径到当前主题目录，彻底杜绝跨主题干扰。
#### 模块核心逻辑（必须严格实现）
1.  **初始化锚点校验**：模块初始化时，自动读取根目录`.current_workspace`锚点文件，获取当前激活主题目录，校验主题目录是否存在，校验不通过直接抛出异常，终止主工作流运行。
2.  **公共路径方法**：实现`get_core_path()`方法，返回`core/`目录绝对路径，仅允许读取，禁止写入。
3.  **主题路径方法**：实现`get_workspace_path(sub_dir="")`方法，传入子目录名称，返回当前主题目录下对应子目录的绝对路径，**主工作流所有写入操作必须使用该方法**。
4.  **写入路径强制校验**：实现`validate_write_path(path)`方法，校验写入路径是否在当前主题目录内，若不在则直接抛出异常，拒绝写入，从代码层面禁止跨主题写入。
5.  **配置读取统一入口**：实现`get_workspace_config(config_name)`方法，统一读取当前主题`config/`目录下的配置文件，返回解析后的配置对象，主工作流不再读取任何根目录的配置。
6.  **记忆库路径方法**：实现`get_memory_path(memory_type)`方法，返回当前主题`memory/`目录下对应记忆库文件路径，确保每个主题有独立的迭代记忆。

**执行要求**：模块开发完成后，校验所有方法逻辑正确，无语法错误，输出路径解析模块开发完成报告，标注核心方法与功能。

## 四、最终交付物要求

## 五、异常处理规则
1.  **文件迁移失败**：自动备份原文件，输出错误日志，不删除原文件，暂停执行，等待用户确认后再继续。
2.  **代码路径适配错误**：自动回滚修改，输出错误详情，逐行修复后重新校验。
3.  **脚本语法错误**：自动修复基础语法问题，无法修复则输出错误详情与行号，不终止整体重构流程。
4.  **测试用例执行失败**：输出详细的失败原因，自动修复可解决的问题，无法解决则明确标注，给出修复方案。
5.  **Git提交失败**：输出报错详情，完成本地提交，不终止整体重构流程，提示用户手动处理远程仓库问题。
