DARIS 项目工程化多主题隔离架构重构
## 一、核心使命
将现有的DARIS单主题项目，彻底重构为**「公共核心库 + 独立工作区隔离」**的工程化架构，实现：
1.  不同研究主题（电力预测、碳排放预测等）的数据、配置、代码、实验、日志、知识库100%隔离，互不干扰
2.  公共工具代码、Prompt模板、基础配置100%复用，避免重复造轮子
3.  提供一键初始化新主题、一键切换主题的工程化脚本
4.  保留现有所有功能，仅重构目录结构与代码组织，不破坏核心逻辑
5.  严格保护用户隐私，.env文件保持全局唯一，不随主题切换

## 二、绝对遵守的强制规则
1.  增量重构：保留现有所有有效代码、配置、数据，仅做目录迁移与结构调整，不删除任何文件
2.  公共/私有分离：通用代码、模板放入core/公共库，主题专属内容放入workspaces/对应目录
3.  向后兼容：重构后，通过切换脚本仍可访问旧主题的所有数据与结果
4.  Git规范：更新.gitignore，确保workspaces/下的大体积数据、日志、模型文件可选择是否提交
5.  隐私保护：.env文件保持在根目录，全局唯一，不随主题切换

对于建立新主题的逻辑理解：
当你运行 python scripts/init_workspace.py carbon 时，系统会按以下步骤完成「主题 ID 和元数据的绑定」，不是只建一个空文件夹：
步骤 1：创建主题专属的隔离工作区
以carbon为唯一 ID，在workspaces/下创建完整的独立目录结构，所有该主题的内容都会存在这里，和其他主题物理隔离：
plaintext
workspaces/workspace_carbon/
├── config/                # 该主题专属配置（含关键词）
├── data/                  # 该主题专属数据集
├── literature/            # 该主题专属文献
├── hypothesis/            # 该主题专属创新点
├── code/                  # 该主题专属定制代码
├── experiment/            # 该主题专属实验结果
├── memory/                # 该主题专属迭代记忆库
├── knowledge_base/        # 该主题专属Obsidian知识库
├── logs/                  # 该主题专属运行日志
└── report/                # 该主题专属最终报告
步骤 2：生成该主题专属的元数据配置模板
系统会从core/config_templates/（公共模板库），把所有配置模板复制到workspace_carbon/config/下，同时生成该主题专属的research_definition.md，里面预留了关键词、研究目标、指标约束等所有个性化配置项，你只需要在这里填写该主题的内容即可。
比如 carbon 主题的config/search_rules.yaml里，你填写的关键词：
yaml
# 仅属于carbon碳排放主题的关键词，其他主题完全看不到
core_keywords: ["碳排放预测", "时空序列预测", "MTGNN 碳排放", "多变量时序预测"]
expand_keywords: ["城市碳排放", "时序分解", "图卷积时序模型"]
exclude_keywords: ["综述", "电力负荷", "能源消耗"]
步骤 3：标记该主题为当前激活状态
系统会自动在根目录的.current_workspace文件里写入workspace_carbon，这个文件是整个 DARIS 工作流的全局锚点，所有环节都会先读这个文件，找到当前激活的主题路径，再读取对应的配置。

整个 DARIS 工作流的所有环节，都会先读取.current_workspace锚点，再定位到对应主题的工作区，读取专属配置，完全不会使用全局配置，实现 100% 隔离。
我按你的工作流环节，拆解每个环节的具体实现：
工作流环节	主题隔离的具体逻辑	关键词的使用方式
文献调研环节	系统先读锚点找到当前主题，再读取该主题config/search_rules.yaml里的关键词，执行检索、下载，文献 PDF 只保存到该主题的literature/pdf/目录	完全使用当前主题专属的关键词，不会用到其他主题的关键词，文献也只存在当前主题的目录里
创新挖掘环节	读取当前主题literature/structured/里的专属文献，生成的创新点、研究假说，只保存到该主题的hypothesis/目录	基于当前主题的文献、研究目标生成创新点，记忆库只读取该主题memory/里的专属迭代记忆
代码实现环节	基线代码复用core/里的公共模型库，定制化代码只写在当前主题的code/目录里，数据集只读取该主题data/目录下的专属数据	代码适配当前主题的数据集、特征规则，不会读取其他主题的数据
实验调优环节	训练配置读取当前主题config/program.md，实验结果、日志、模型只保存到该主题的experiment/、logs/目录	调优目标、指标约束、迭代规则，完全使用当前主题的专属配置
结果沉淀环节	报告、知识库、迭代记录，只同步到当前主题的report/、knowledge_base/目录	完全基于当前主题的实验结果生成报告，和其他主题完全隔离
